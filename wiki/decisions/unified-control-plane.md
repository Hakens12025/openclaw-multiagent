# 统一控制面

> 单一控制面：graph auth + ingress.normalize + conveyor.dispatch + lifecycle.commit + loop engine。六条独立分发路径收敛为一。

## 决策

建立单一控制面，所有消息/任务流经同一条路径：
1. **graph auth** — 权限校验
2. **ingress.normalize** — 入口标准化
3. **conveyor.dispatch** — 统一分发
4. **lifecycle.commit** — 生命周期提交
5. **loop engine** — 环路检测与管理

此前 5 条共存的分发路径、4 个并行的 busy/queue 状态容器、3 条回流路径全部收敛。

## 原因

系统存在三种反模式：
- **半统一**：只统一 dispatch 不统一 lifecycle — 入口一样但语义分裂。
- **假统一**：同一入口但各自维护事实源 — 看似统一实则各说各话。
- **绕过统一**：主线统一但 loop/prompt/test 走私有协议 — 测试和生产行为不一致。

5 条分发路径 + 4 个状态容器 + 3 条回流路径 = "胶水系统"，bug 的温床。

## 红线

- 不为重构造新轮子
- 净新增代码为负数
- 不加兼容层
- 不加新协议
- 不加新状态容器
- 不加新 timer owner

## 否决的替代方案

1. **只统一 dispatch，保留多套私有 lifecycle 语义** — 半统一反模式。
2. **把 loop 当作独立协议** — 绕过统一反模式。
3. **备忘录90的加法式重构** — 违反"净新增为负"原则。
