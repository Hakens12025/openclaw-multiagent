# Wiki Schema — OpenClaw Knowledge Base

> 本文件定义 wiki 的维护规则。LLM 在操作 wiki 前必须读此文件。

## 定位

```
代码 (extensions/watchdog/...)  → 拥有 WHAT（当前实现）
备忘录 (use guide/)             → 拥有 RAW（讨论原始记录，write-once）
wiki (wiki/)                    → 拥有 WHY（设计意图、决策、演化脉络，write-many）
```

wiki 不复述代码做了什么，不修改备忘录，不替代 CLAUDE.md 的硬规则。
代码和 wiki 冲突时，代码赢 — wiki 必须更新。

## 目录结构

```
wiki/
  schema.md         ← 本文件（wiki 规则）
  index.md          ← 全局索引（按主题分类，单行摘要）
  log.md            ← 操作日志（append-only）
  status.md         ← 当前项目状态（活跃任务、阻塞、下一步）
  concepts/         ← 概念页（系统里的"名词"）
  decisions/        ← 决策页（为什么这么做、为什么不那么做）
```

## 页面格式

### 概念页 (concepts/)

```markdown
# {概念名}

> 一句话定义

## 是什么
简明描述，不超过 3 段。引用代码位置（`lib/routing/dispatch-graph-policy.js`）但不复制代码。

## 为什么存在
解决什么问题，没有它会怎样。

## 和谁交互
列出关联概念，用 wiki 内链 `[概念名](../concepts/xxx.md)`。

## 演化
按时间记录关键变化。格式：`v{N}-stable: 发生了什么`。
引用备忘录出处：`源: 备忘录90 §一`。

## 当前状态
一句话：稳定 / 演化中 / 待重构 / 已废弃。
```

### 决策页 (decisions/)

```markdown
# {决策标题}

> 一句话结论

## 决策
做了什么选择。

## 原因
为什么这么选。

## 否决的替代方案
考虑过什么、为什么不用。每个替代方案附带否决理由。

## 影响
这个决策影响了哪些概念/模块。

## 出处
`源: 备忘录{N} §{章节}` 或 `讨论日期: YYYY-MM-DD`
```

### status.md

自由格式，但必须包含：
- 当前在做什么（活跃 initiative）
- 做到哪了（checkbox todo）
- 阻塞什么
- 下一步是什么

## 三个操作

### Ingest（编译）

触发：写完备忘录后，或讨论达成共识时。

流程：
1. 识别备忘录/讨论中的新知识点
2. 对每个知识点：已有概念页 → 更新；没有 → 创建
3. 检查是否产生新决策 → 写决策页
4. 更新所有被影响的页面的交叉引用
5. 更新 index.md
6. 追加 log.md

编译 ≠ 复制。提取结论和原因，丢弃讨论过程。

### Query（查询）

新 session 启动时：
1. 读 index.md → 定位相关概念页
2. 读概念页获取上下文
3. 如果查询产生了新知识，反写为新 wiki 页

### Lint（健康检查）

定期或主动触发：
- 概念页引用的代码位置还存在吗？
- 两个页面对同一事实有矛盾描述吗？
- 有孤页（没有任何其他页面链接到它）吗？
- 有概念被多次提及但没有自己的页面吗？
- status.md 里的 todo 和实际进度一致吗？

## 规则

1. **wiki 页面必须自包含** — 读一个页面就能理解该概念，不需要跳转 5 个地方
2. **引用不复制** — 指向代码文件和备忘录，不把内容搬过来
3. **交叉引用用相对链接** — `[三层协议](../concepts/three-layer-protocol.md)`
4. **每次更新 wiki 必须同步更新 index.md 和 log.md**
5. **代码位置用格式** `lib/xxx.js` — 如果文件改名了，lint 会发现
6. **不要写百科全书** — 概念页控制在 200 行以内，超了就该拆
